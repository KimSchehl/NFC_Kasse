## What This Project Is

**NFC-Kasse** is a cashless POS system for events. Guests receive NFC wristbands with a pre-loaded balance; staff scan wristbands on a tablet to deduct purchases. The system runs on a local LAN (no internet required at runtime).

- **Backend:** FastAPI + SQLite, running on a Windows thin client
- **Frontend:** Flutter tablet app (Android), deployed to up to 4 tablets via ADB Wi-Fi
- **NFC Input:** Native `nfc_manager` on Android; USB HID keyboard emulation on desktop/web
- **Language convention:** Code and DB identifiers in English; all user-facing text in German

All files live under `NFC_Flutter_Project/`:

```
NFC_Flutter_Project/
├── backend/              # Python FastAPI backend
├── nfc_kasse_app/        # Flutter app
├── docs/                 # Architecture docs, use-case walkthroughs, API overview

├── start_backend.sh      # Linux/macOS launcher
└── build_and_deploy.bat  # Build APK + ADB deploy to tablets
```

## Architecture

### Backend

**Entry point:** `main.py` — registers routers, configures CORS.

**Database access:** `database.py` provides `get_db(exclusive=False)` context manager. All booking and payout operations must use `get_db(exclusive=True)` to get a SQLite `BEGIN EXCLUSIVE` lock and prevent concurrent balance races.

**Auth pipeline:**
1. `dependencies.py` / `get_current_user()` — validates the JWT Bearer token on every request
2. `require_permission("permission.id")` — FastAPI `Depends` factory for endpoint-level permission checks
3. Refresh tokens are stored SHA-256 hashed in `refresh_token` table; on refresh the old token is revoked and a new pair is issued (token rotation)

**Routers and their core responsibilities:**

| Router | Endpoints | Key logic |
|---|---|---|
| `auth.py` | `/login`, `/refresh`, `/logout`, `/me` | JWT + refresh token lifecycle; `/me` returns full permission set + category access flags for the session |
| `products.py` | `/categories`, `/products` | Category list filtered by `user_category_access`; managers see all |
| `sales.py` | `/balance/{uid}`, `/` (POST booking), `/{id}/cancel` | Exclusive transaction; per-category `can_book` / `can_storno_*` checks |
| `topup.py` | `/topup`, `/payout` | Requires `guthaben.topup` / `guthaben.payout` global permission |
| `users.py` | `/users`, `/users/{id}/permissions`, `/users/{id}/categories` | User CRUD + fine-grained permission + category-access assignment |
| `stats.py` | `/revenue`, `/transactions`, `/export` | Revenue rollup, transaction history, CSV export |

### Permission Model

**Two layers:**

1. **Global permissions** (`user_permission` table) — coarse capabilities:
   - `guthaben.topup`, `guthaben.payout`
   - `categories.create`, `categories.edit`, `categories.deactivate`, `categories.delete`
   - `statistics.revenue`, `statistics.transactions`, `statistics.export`
   - `users.view`, `users.create`, `users.edit`, `users.deactivate`, `users.manage_permissions`

2. **Per-category permissions** (`user_category_access` table) — 7 boolean columns:
   - `can_book`, `can_storno_5min`, `can_storno_unlimited`
   - `can_create_article`, `can_edit_article`, `can_deactivate_article`, `can_delete_article`

**Manager bypass:** Any user with at least one of `categories.create / edit / deactivate / delete` is treated as a manager — they bypass all per-category checks and see every category with full flags. Check is done with `_is_manager(db, user_id, event_id)` in each router that needs it; `/me` sets `is_manager: true` in the response.

**Storno exclusivity (UI only):** `can_storno_5min` and `can_storno_unlimited` are mutually exclusive in the edit dialog — selecting one clears the other. The backend treats them as independent booleans (both are valid independently).

### Flutter App

**State management (Riverpod):**

- `authProvider` (`AsyncNotifierProvider`) — auth lifecycle; on cold start silently restores session via `/me` before showing the login screen
- `cartProvider` (`NotifierProvider<CartNotifier, List<CartItem>>`) — in-memory cart; `addProduct` merges quantities
- `categoriesProvider` / `productsProvider` — `FutureProvider(.family)`; invalidated via `categoriesRefreshProvider` / `productsRefreshProvider` integer bump
- `currentScreenProvider` / `selectedCategoryProvider` / `editModeProvider` — navigation and POS UI state

All providers are defined in `lib/providers/providers.dart`.

**API client (`lib/services/api_client.dart`):**

Dio instance with a request interceptor that injects the Bearer token and a response interceptor that catches 401, calls `/refresh` once (shared `_refreshFuture` prevents concurrent refresh storms), updates stored tokens, and retries the original request.

**Responsive layout:**
- `≥ 600 px`: persistent `AppSidebar` + content area (tablet landscape)
- `< 600 px`: `Drawer` + full-screen content (phone)

**Edit dialogs (`lib/widgets/dialogs/`):**

- `edit_user_dialog.dart` — the most complex widget; renders a tri-state checkbox tree for both global permissions (`_PermissionGroup`) and per-category permissions (`_CategoryTreeTile`). The storno section uses a real `Radio` widget pair wrapped in `RadioGroup` (Flutter ≥ 3.32 API).
- `edit_product_dialog.dart` — includes `isPayout` switch (marks article as payout-type; full payout behavior is Phase 2).
- `edit_category_dialog.dart` — rename + optional delete (requires `categories.delete` global permission).

## Database Schema Highlights

Key tables and non-obvious design decisions:

- **`sale`** — append-only; cancellation sets `cancelled = 1` (never deleted). This ensures an immutable audit trail.
- **`product`** — `price` is stored in cents as integer; negative prices are refunds. `is_payout` flag marks an article whose booking should payout the full remaining balance (Phase 2 logic not yet implemented).
- **`customer`** — `is_available` column added (DEFAULT 1) for chip tracking (Phase 2, not yet wired to business logic).
- **`user_category_access`** — replaces the old flat `sales.*` permissions. One row per (user, event, category) with 7 fine-grained boolean columns.
- **`permission_node`** — the permission tree is data-driven (seeded in `init_db.py`), not hardcoded in router logic.

## Open Work (Phase 2)

- **Payout article behavior:** When a product with `is_payout = true` is booked, the booking price should equal the customer's current balance (i.e., a full payout). This is not yet implemented in `sales.py`.
- **`is_available` customer flag:** DB column exists but no business logic reads or writes it beyond the schema.
- **Multi-event support:** Schema supports multiple events per tenant; the UI and backend currently hard-code `event_id = 1`.
- **Update mechanism:** `update.py` / `update_dialog.dart` are scaffolded but not fully wired.
